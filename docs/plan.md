# AI 开发总目标

基于以下技术栈完成一个可运行的 MVP：

- **前端**：Vue 3 + Element Plus
- **后端**：FastAPI
- **数据库**：MySQL
- **缓存**：Redis
- **功能范围**：
  - 商品搜索
  - 商品详情
  - 多平台比价
  - 历史价格展示
  - 优惠信息展示
  - 决策建议
  - 降价提醒

---

## 一、开发原则

先让 AI 做对，再让 AI 做快。

### 整体原则

1. 先骨架，后业务
2. 先跑通，后优化
3. 先假数据，后真采集
4. 一次只做一个明确目标
5. 每一步都要求 AI 输出可运行结果

> [!TIP]
> 你给 AI 的任务，尽量不要一句话说“把整个项目做完”，而要拆成阶段。

## 三、最适合丢给 AI 的执行方式

不要一次让 AI 做全部，而是按下面模式下发：

### 模式 1：单任务执行

每次只给一个任务，例如：

**示例提示词**：
你是资深全栈工程师，请在当前项目中完成“FastAPI 商品搜索接口”。

要求：
1. 使用现有技术栈 FastAPI + SQLAlchemy + MySQL
2. 实现 `GET /api/v1/search?q=xxx&page=1&page_size=20`
3. 返回商品主信息和最低价
4. 按 service / repository / schema / router 分层
5. 生成可运行代码，不要只给示例
6. 包含必要的错误处理和注释
7. 修改完成后说明新增了哪些文件、改了哪些文件、如何验证

---

### 模式 2：严格约束输出

每次都要求 AI 按这个顺序输出：
1. 先说明本次改动目标
2. 再列出要新增/修改的文件
3. 再给出完整代码
4. 再给出运行命令
5. 再给出验证步骤
6. 再列出后续待办

这样 AI 不容易乱。

## 六、你最该避免的 4 个问题

1. **不要一句话让 AI 做完整项目**：这样大概率会目录乱、命名乱、代码重复、跑不起来。
2. **不要先做爬虫**：先用假数据跑通产品，再接采集。
3. **不要一开始追求“完美架构”**：MVP 阶段先能跑。
4. **不要不验收就进入下一步**：每一阶段都要能运行。
